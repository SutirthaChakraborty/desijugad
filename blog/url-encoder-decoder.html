Why URLs Cannot Contain Arbitrary Characters

A URL is transmitted as plain text across systems that were originally designed for ASCII. Spaces, accented letters, symbols like &, =, and #, and any non-ASCII characters (Arabic, Hindi, Chinese script) carry special meaning in a URL's structure or simply cannot be transmitted safely. URL encoding — formally called percent-encoding — solves this by converting every unsafe character into a % followed by its two-digit hexadecimal ASCII or UTF-8 byte value.

What Gets Encoded and Why

RFC 3986, the specification that defines URI syntax, divides characters into two categories:

  • Unreserved characters — letters (A–Z, a–z), digits (0–9), hyphen (-), underscore (_), period (.), and tilde (~). These are safe to use as-is anywhere in a URL.
  • Everything else must be percent-encoded when used as data rather than as URL structure.

Key examples and the reason each matters:

  • Space → %20 (or + in query strings). Spaces are not valid in URLs at all — without encoding, many parsers treat the space as the end of the URL.
  • & → %26. The ampersand separates query parameters (?a=1&b=2). If your parameter value contains an ampersand, it must be encoded or the parser splits the value in two.
  • # → %23. The hash marks the start of a URL fragment. If you include a literal # in a query value without encoding, everything after it is treated as the fragment and never sent to the server.
  • Non-ASCII characters (e.g., Hindi नमस्ते) are first encoded as UTF-8 bytes, then each byte is percent-encoded. The result is a sequence like %E0%A4%A8%E0%A4%AE%E0%A4%B8%E0%A5%8D%E0%A4%A4%E0%A5%87.

URL Encoding vs Base64

These two encoding schemes are often confused but serve entirely different purposes:

  • URL encoding (percent-encoding) preserves the readability of ASCII text while making it safe for use inside URL components. hello world becomes hello%20world — still legible to a human.
  • Base64 converts arbitrary binary data into a string of printable ASCII characters using a 64-character alphabet. The result is not human-readable. Use Base64 when you need to embed binary data — like a small image — inside a URL (data URIs) or in a JSON payload.

Rule of thumb: use percent-encoding for query string parameter values; use Base64 for binary blobs that need to travel through text-only channels.

Practical Examples

  • Search query: The search term "hello world" in a query string becomes either q=hello%20world or q=hello+world depending on whether you use strict RFC 3986 encoding or the older HTML form encoding (the + convention comes from application/x-www-form-urlencoded).
  • API key with special characters: If an API key contains +, /, or = (common in Base64-encoded keys), these must be percent-encoded before being placed in a query string. Failing to do so causes the server to receive a mangled key.
  • Decoding a webhook URL: Third-party services often send callback URLs that look like opaque strings. Decoding the percent-encoding reveals the actual parameters and values — useful for debugging what a service is actually sending.
  • Debugging a split parameter: If a URL parameter like redirect_url=https://example.com/path?foo=bar is passed unencoded, the outer query parser sees two parameters: redirect_url=https://example.com/path?foo and bar. Encoding the value first prevents this: redirect_url=https%3A%2F%2Fexample.com%2Fpath%3Ffoo%3Dbar.

encodeURIComponent vs encodeURI in JavaScript

JavaScript ships with two built-in encoding functions, and choosing the wrong one is a common bug:

  • encodeURI(url) encodes most special characters but deliberately leaves the structural characters of a URL untouched: :, /, ?, #, &, =, @. Use this when you have a complete URL and want to make it safe to transmit without breaking its structure.
  • encodeURIComponent(value) encodes everything including those structural characters. Use this on individual parameter values before assembling them into a query string. Example: encodeURIComponent("a=1&b=2") produces "a%3D1%26b%3D2" — safe to embed as a value without it being parsed as extra parameters.

Common Pitfalls

  • Double-encoding: If you encode an already-encoded URL, %20 becomes %2520 (because % itself gets encoded to %25). Always decode before re-encoding if you are unsure of the input state.
  • Forgetting to decode before display: Showing a raw encoded URL to a user — especially one containing non-ASCII script — gives a terrible experience. Decode before rendering in UI.
  • + vs %20: HTML form submissions use + for spaces; manual URL construction and most server-side URL parsers expect %20. PHP's urldecode() handles both, but JavaScript's decodeURIComponent() does not decode + as a space — use .replace(/\+/g, ' ') first if dealing with form-encoded data.

Frequently Asked Questions

What is the difference between + and %20 for spaces?

Both represent a space character, but in different contexts. %20 is the strict RFC 3986 percent-encoding of a space and is universally valid in any part of a URL. The + sign as a space substitute comes from the application/x-www-form-urlencoded format — the encoding used by HTML forms when submitted via GET or POST. Most server-side frameworks decode both, but when building URLs manually or using decodeURIComponent() in JavaScript, + is treated as a literal plus sign, not a space. To be safe in programmatic URL construction, always use %20.

Why do some URLs look like %E0%A4%A8%E0%A4%AE%E0%A4%B8%E0%A5%8D%E0%A4%A4%E0%A5%87?

That sequence is a Hindi word (नमस्ते, meaning "hello") encoded as UTF-8 bytes, with each byte percent-encoded. Non-ASCII characters cannot be represented directly in a URL, so they are first converted to their UTF-8 byte representation (which can be 2–4 bytes per character for most scripts), and then each byte is written as %XX where XX is the hexadecimal value. Modern browsers display the decoded Unicode form in the address bar for readability, but the actual HTTP request sends the percent-encoded version.

Is URL encoding the same as HTML entity encoding?

No. They solve similar problems in different contexts. URL encoding (percent-encoding) makes characters safe for use inside a URL — it uses %XX notation. HTML entity encoding makes characters safe for display inside HTML markup — it uses &amp;, &lt;, &gt;, &#160; and similar forms. An ampersand in a URL query parameter needs to be percent-encoded (%26); an ampersand in HTML text needs to be entity-encoded (&amp;). When a URL is embedded in an HTML attribute like href, you often need both: the URL itself is percent-encoded, and then the resulting string is entity-encoded for the HTML context.
