Base64 encoding and decoding: what it is, why it exists, and how to use it

9 min read · Updated April 2026

You've seen it: a long string of seemingly random letters, numbers, and +/= characters in a config file, an email header, or a CSS stylesheet. That's Base64. It's one of those things every developer eventually encounters, usually in a confusing context, and has to look up. This guide explains it from scratch.

The problem Base64 solves

Binary data — images, audio, compiled code, encrypted payloads — contains every possible byte value from 0 to 255. Text-based systems, historically, were designed to handle only a subset of those values: printable ASCII characters, which occupy byte values 32–126.

When binary data had to travel through text-based channels — early email systems (SMTP), HTML attributes, JSON fields, HTTP headers — bytes outside the printable ASCII range caused problems: they got stripped, corrupted, or misinterpreted as control characters.

Base64 is the solution: a way to represent any sequence of bytes using only 64 safe characters (A–Z, a–z, 0–9, +, /, and = for padding). It converts arbitrary binary to text that can safely travel through any text channel.

How it works (in plain language)

Base64 takes every 3 bytes of input (24 bits) and represents them as 4 characters from its 64-character alphabet (6 bits per character, 4 × 6 = 24 bits). Each output character represents a number from 0–63, mapped to a letter or digit.

Example:

Text input:   "Man"
ASCII bytes:  77, 97, 110
Binary:       01001101 01100001 01101110
6-bit groups: 010011 010110 000101 101110
Decimal:      19      22     5      46
Base64 chars: T       W      F      u
Output:       "TWFu"

If the input isn't divisible by 3, = padding characters are added to make the output length a multiple of 4.

The full specification is in RFC 4648 — The Base16, Base32, and Base64 Data Encodings published by the IETF.

Base64 is encoding, not encryption

This is the single most important thing to understand about Base64: it is trivially reversible by anyone. There is no key, no secret, no security. Encoding "password123" as Base64 does not hide it — anyone who sees the Base64 string can decode it instantly.

Base64 is a transport and storage encoding. It makes binary safe for text channels. It does not protect data.

If you see credentials, tokens, or passwords stored in Base64 in a config file or code repo, that is not security — it's cosmetic obfuscation. The data is effectively in plaintext.

Where Base64 is actually used

Email attachments (MIME)

SMTP, the email protocol, was designed for ASCII text. Attachments — PDFs, images, Office files — are binary. The MIME standard (RFC 2045) specifies that binary attachments are Base64-encoded in the email body. Every email client and server handles this transparently.

Data URIs in HTML/CSS

You can embed images directly in HTML or CSS as Base64 data URIs, avoiding a separate HTTP request:

<img src="data:image/png;base64,iVBORw0KGgoAAAANSUhEUgAA..." />
background-image: url('data:image/svg+xml;base64,PHN2ZyB4bWxucy...');

This is useful for small icons and loading spinners to eliminate extra HTTP requests. The tradeoff: the data is ~33% larger than the binary equivalent, and the data URI is not separately cacheable.

JWT (JSON Web Tokens)

JWT tokens — used for authentication in virtually every modern web app — are three Base64URL-encoded JSON objects joined by dots. If you see a token like eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJ1c2VyIn0.abc123, you can decode the first two parts in any Base64 decoder and read the header and claims in plaintext. Again: not encryption, encoding.

Storing binary data in text formats

JSON doesn't have a native binary type. Databases that store JSON (MongoDB, PostgreSQL JSONB) represent binary fields as Base64 strings. The same is true for XML. Many APIs return images, files, or cryptographic material as Base64 strings in JSON responses.

API credentials and tokens

HTTP Basic Authentication encodes username:password in Base64 in the Authorization header: Authorization: Basic dXNlcjpwYXNz. This is only safe over HTTPS (which decrypts the connection) — Base64 itself provides no protection.

Base64 vs Base64URL

Standard Base64 uses + and / as the 62nd and 63rd characters. In URLs, + means space and / is a path separator — both cause problems. Base64URL replaces them with - and _, and omits padding. You'll see Base64URL in JWTs, URL tokens, and file names. Most decoders handle both; MDN's Base64 reference covers both variants.

How to encode/decode in code

JavaScript (browser & Node.js)

// Encode
btoa("Hello, world!")  // "SGVsbG8sIHdvcmxkIQ=="

// Decode
atob("SGVsbG8sIHdvcmxkIQ==")  // "Hello, world!"

// For binary/unicode, use TextEncoder + Uint8Array
const encoded = btoa(String.fromCharCode(...new TextEncoder().encode("Hello")));

See the MDN btoa() reference for edge cases with Unicode strings.

Python

import base64

# Encode
base64.b64encode(b"Hello, world!")  # b'SGVsbG8sIHdvcmxkIQ=='

# Decode
base64.b64decode("SGVsbG8sIHdvcmxkIQ==")  # b'Hello, world!'

# File to Base64
with open("image.png", "rb") as f:
    encoded = base64.b64encode(f.read()).decode("utf-8")

Command line

# Encode
echo -n "Hello" | base64          # SGVsbG8=
base64 image.png > image.b64      # File to Base64

# Decode
echo "SGVsbG8=" | base64 --decode  # Hello
base64 --decode image.b64 > image_out.png

Other popular Base64 tools

  • Base64Encode.org — server-based, widely used, clean UI.
  • Base64Decode.org — companion decoder site.
  • CyberChef (GCHQ) — the most powerful free encoding/decoding tool online, open-sourced by GCHQ. Runs in browser. Handles Base64, hex, rot13, XOR, hashing, and hundreds of other operations. Indispensable for developers and security researchers.
