Diff Checker Online — Compare Text and Highlight Differences Free

April 27, 2026 • 4 min read

Why Text Comparison Is a Fundamental Developer Skill

Every developer eventually faces the same problem: two versions of something — a config file, a script, a document — and no clear picture of what changed between them. The ability to compare text precisely is not a nice-to-have; it sits at the core of code review, debugging, deployment verification, and collaborative writing. Understanding how diff tools work under the hood makes you far more effective when reading their output.

How a Diff Algorithm Works

Most diff tools — including git diff — use a variant of the Myers diff algorithm, published by Eugene Myers in 1986. The core idea is finding the Longest Common Subsequence (LCS): the longest sequence of lines (or characters) that appears in both texts in the same order, even if they are not contiguous.

Once the LCS is computed, every line in the original text that is not part of the LCS is marked as removed (shown with a - prefix), and every line in the new text that is not part of the LCS is marked as added (shown with a + prefix). Lines that are part of the LCS appear unchanged — these are the context lines, typically shown in groups of three on each side of a change to help you understand what surrounds each edit.

Two display formats exist for diffs:

  • Unified diff — both versions shown in a single column, with - and + prefixes. This is the format git diff produces and what patch files use.
  • Side-by-side diff — original on the left, modified on the right. Easier to read for large changes because your eyes can track both versions in parallel.

Reading a Unified Diff: Decoding the @@ Header

The most confusing part of a raw diff output is the hunk header. Consider:

@@ -3,7 +3,8 @@

Breaking it down:

  • -3,7 — in the original file, this hunk starts at line 3 and spans 7 lines.
  • +3,8 — in the new file, the corresponding section starts at line 3 and spans 8 lines.

The difference in line count (7 vs 8) tells you that one line was added. Context lines are counted in both numbers, which is why even unchanged surrounding lines are included in the span. If both numbers were equal, the same number of lines were added as were removed.

Context lines serve a practical purpose: they anchor the change in the file so a patch tool knows exactly where to apply modifications even if line numbers have shifted from earlier insertions.

Real-World Use Cases for Diff

  • Code review: Before merging a pull request, reviewers compare the feature branch against the main branch to understand precisely what logic changed — not just what files changed.
  • Config file audits: Infrastructure teams diff nginx.conf or docker-compose.yml between environments to verify that staging and production only differ where expected.
  • Document drafts: Editors comparing a revised manuscript against the previous version to verify that only the agreed changes were made — no accidental deletions.
  • Debugging regressions: "It worked yesterday." Running diff against the last known-good version of a config or script narrows the cause from "something changed" to "exactly this changed."
  • Verifying patches: Patch files (.patch or .diff) are applied to source trees in open-source projects. Reading them in a diff viewer confirms what a patch will do before you apply it.

Diff in a Git Workflow

Git exposes diff at several stages of the development lifecycle:

  • git diff — shows changes in your working tree that have not been staged yet.
  • git diff --staged — shows changes that have been staged (added to the index) but not yet committed. Also written as git diff --cached.
  • git diff HEAD~1 — compares the current working state against the previous commit. Useful for reviewing what you just changed before pushing.
  • git diff branch1..branch2 — compares the tip of two branches. Useful for reviewing what a feature branch adds relative to main.

Pull request interfaces on GitHub and GitLab take this same underlying diff and render it visually — coloring additions green and removals red, collapsing unmodified files, and linking line numbers to inline comments. The visual layer is a convenience; the data underneath is always a standard unified diff.

Practical Tips

  • Ignore whitespace: diff -w (or git diff -w) skips whitespace-only changes. Invaluable when a file has been reformatted but no logic changed.
  • Word-level diffs: git diff --word-diff highlights individual changed words within a line rather than marking the whole line as changed. Useful for prose and long config values.
  • Binary file limitations: Standard diff tools operate on text. Binary files (images, compiled executables, PDFs) show only "binary files differ." Specialised tools exist for image diffs and document comparison, but they work on extracted text layers, not bytes.

Frequently Asked Questions

What is the difference between unified and side-by-side diff?

A unified diff shows both the old and new versions in a single column. Removed lines are prefixed with - and added lines with +. It is compact and the standard format for patch files and git diff output. A side-by-side diff displays the original file in the left column and the modified file in the right column, with matching lines aligned. Side-by-side is generally easier to read for large or complex changes because you can scan both versions simultaneously without mentally reconstructing the modified file.

Can I diff binary files or images?

Standard line-based diff tools are designed for text and cannot meaningfully compare binary content — they will simply report "binary files differ." For images, specialised diff tools can overlay two versions and highlight pixel-level differences, or extract text from PDFs before comparing. For binary formats that have a text representation (like SVG, which is XML, or a compiled binary where you compare the disassembly), converting to text first and then diffing is the standard workaround.

What does "hunk" mean in a diff?

A hunk is a contiguous block of changes in a diff output. If you changed line 5 and line 50 of a file, those are two separate hunks. Each hunk starts with an @@ header indicating the line ranges it covers, followed by context lines and the actual additions and removals. Tools like git add -p (interactive staging) let you stage or skip individual hunks, which is why the term appears frequently in interactive Git workflows.
